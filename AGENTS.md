# CellCom CRM — Agent Guide

## Project Overview

CellCom CRM is a full-stack CRM application targeting Malaysian SMEs. It is built on **SpacetimeDB** (a realtime relational database with server-side logic) and a **React + TypeScript** client. The core data model includes a **Knowledge Graph** (`kg_vertex` / `kg_edge`) that auto-syncs from operational tables to represent entities and their relationships.

The project uses a client-server architecture where the backend is a SpacetimeDB TypeScript module (not a traditional HTTP API), and the frontend communicates with it via generated TypeScript bindings over WebSocket.

## Build and Development Commands

### Prerequisites

- Node.js ≥ 18
- SpacetimeDB CLI ≥ 2.1

### Start SpacetimeDB locally

```bash
spacetime start
```

### Build and publish the server module

```bash
cd server/spacetimedb
spacetime build
spacetime publish cellcomcrm   # or the database name from spacetime.local.json
```

### Install and run the client

```bash
cd client
npm install
npm run dev        # Vite dev server on http://localhost:5173
```

### Build client for production

```bash
cd client
npm run build      # Outputs to client/dist/
npm run preview    # Preview production build locally
```

### Regenerate client bindings

Whenever the server schema or reducers change, run:

```bash
./generate-bindings.sh
```

This builds the server module and runs `spacetime generate --lang typescript` into `client/src/generated/`.

## Runtime Architecture

1. **SpacetimeDB** runs as the central backend (local or cloud). It holds all tables and executes reducers (transactional stored procedures).
2. The **React client** connects via WebSocket using the generated `DbConnection` class.
3. On connect, the client subscribes to `SELECT * FROM <table>` for every table so the local client DB mirrors the server.
4. All mutations go through **reducers** (e.g., `createContact`, `updateDeal`) — never direct table inserts from the client.
5. The Knowledge Graph is maintained **server-side** via dual-write helpers (`syncKgVertex`, `createKgEdge`, `deleteKgVertex`) inside reducers.

## Data Model Conventions

### Currency
All monetary values are stored as **u64 cents**.
- Example: `RM 100.50` is stored as `10050n`.
- The client formats with `toLocaleString('en-MY')`.

### JSON Fields
SpacetimeDB 2.1 TypeScript modules do not have a native JSON type. JSON objects are stored as **strings**.
- The client must `JSON.stringify` before sending and `JSON.parse` after receiving.
- Fields affected: `contacts.custom_fields`, `companies.address`, `companies.billing_address`, `messages.attachments`, `kg_vertex.properties`, `kg_edge.properties`, `workflows.trigger_config`, `workflows.steps`, etc.

### Vector Embeddings
`kg_vertex.vector_embedding` is an optional `f32[]`. There is no fixed DB constraint; **384 dimensions** are expected by convention.

### Timestamps
All `created_at` / `updated_at` fields use SpacetimeDB `t.timestamp()` ( server-generated).

### Enums
SpacetimeDB enums are represented as `{ tag: 'VariantName' }` objects in TypeScript. Always use `.tag` for comparison or display.

## Code Organization

### Server (`server/spacetimedb/src/index.ts`)

The server is a single-file module (~1600 lines) organized into sections:

1. **Enums** — `UserRole`, `ContactSource`, `DealStatus`, `ActivityType`, `ChannelType`, `InvoiceStatus`, `PaymentMethod`, `EntityType`, `RelationType`, etc.
2. **Operational Tables** — `users`, `contacts`, `companies`, `pipelines`, `pipeline_stages`, `deals`, `activities`, `conversations`, `messages`, `products`, `invoices`, `invoice_items`, `payments`
3. **Knowledge Graph Tables** — `kg_vertex`, `kg_edge`
4. **Auth Table** — `tenant_member`
5. **Workflow Tables** — `workflows`, `workflow_executions`
6. **Schema export** — `export default spacetimedb`
7. **KG sync helpers** — `syncKgVertex`, `deleteKgVertex`, `createKgEdge`, `findKgVertexId`
8. **Reducers** — grouped by domain (User, Contact, Company, Pipeline, Deal, Activity, Conversation, Message, Product, Invoice, Payment, Workflow)
9. **Seed data** — `seedDemoData` reducer populates tenant `1n` with demo companies, contacts, deals, stages, products, and activities

### Client (`client/src/`)

- **`spacetime/client.ts`** — Builds the `DbConnection`, sets host/DB name from `import.meta.env`, and defines the subscription query list. It also auto-calls `seedDemoData` if the contacts table is empty.
- **`spacetime/hooks.ts`** — Thin wrappers around `spacetimedb/react` hooks.
- **`spacetime/useKg.ts`** — Filters vertices/edges by tenant and optional entity type.
- **Components** — Each major page is a component. `Layout.tsx` provides the sidebar navigation and topbar.

## Testing

There is **no test framework** currently configured in this project. If you add tests, prefer placing them alongside source files or in a `__tests__` directory. Both client and server use strict TypeScript, which catches many errors at compile time.

## Security Considerations

- **Authentication is disabled in demo mode.** The `requireTenant()` function is currently a no-op. The `tenant_member` table exists for future multi-tenant auth.
- All reducers still receive `tenant_id` and call `requireTenant(ctx, tenant_id)` so that enabling auth later is a drop-in replacement.
- Do not store secrets in `client/.env` — the client bundle is public. `VITE_SPACETIME_HOST` and `VITE_SPACETIME_DB` are safe because they are runtime configuration.
- The server module is compiled to WASM and published to SpacetimeDB; treat the source as the single source of truth for data integrity.

## Deployment Notes

- **SpacetimeDB module**: Publish via `spacetime publish` to either a local node or SpacetimeDB Cloud (`maincloud`). Configs live in `server/spacetime.json` (cloud) and `server/spacetime.local.json` (local).
- **Client**: The client is a static SPA. Build with `npm run build` and serve the `client/dist/` folder from any static host (e.g., Vercel, Netlify, Nginx, S3).

## Common Pitfalls for Agents

1. **Do not edit `client/src/generated/` directly.** Always change the server module and run `./generate-bindings.sh`.
2. **JSON fields are strings.** Remember to stringify before passing to reducers and parse when reading from tables.
3. **Enum comparisons must use `.tag`.** Example: `contact.status.tag === 'Lead'`.
4. **BigInt literals need the `n` suffix.** Example: `tenant_id: 1n`.
5. **Currency is in cents.** Divide by 100 and format when displaying to users.
6. **The KG does not replace operational tables.** Always mutate operational tables via reducers; the KG syncs automatically.
7. **No traditional REST API.** All data access goes through SpacetimeDB subscriptions and reducers.
